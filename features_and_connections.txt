1. PROJECT OVERVIEW
================================================================================

Headwind MDM is a Mobile Device Management (MDM) solution for Android devices.
It consists of 4 main components:

A) hmdm-android (APK) - Android MDM Agent App
   - Package: com.hmdm.launcher
   - Min SDK: 26 (Android 8.0+)
   - Target SDK: 34
   - Type: Device Owner / Device Administrator app
   
B) hmdm-server - Java Backend Server (Maven project)
   - Technology: Java, REST API, PostgreSQL
   - Port: 8080
   - Deployed at: https://hmdm-server-qlw2.onrender.com

C) hmdm-data-api - Node.js Data API Server
   - Technology: Express.js, JWT Auth, MongoDB
   - Deployed at: https://hmdm-data-apicallandcontact.onrender.com

D) hmdm-docker - Docker Deployment Configuration
   - Services: PostgreSQL, Certbot (SSL), HMDM Server
   
E) Google Apps Script - Sheets Integration
   - URL: https://script.google.com/macros/s/AKfycbzTgOLsjh1mluJwXVgySq-zJXO6-drWr8g_JWMfZxSqMkGlHKjtvKwDVp-8z8VkQftetQ/exec

Data Flow:
1. APK connects to hmdm-server for:
   - Device enrollment & configuration
   - Remote commands (reboot, password reset, lock)
   - Push notifications (MQTT / Long Polling)
   - Remote logging
   - Location tracking
   - Application management

2. hmdm-server connects to PostgreSQL for:
   - Device data storage
   - Configuration storage
   - User accounts
   - Application metadata

3. hmdm-server connects to hmdm-data-api for:
   - Contact sync data
   - Call log sync data
   - Notification data

4. Google Apps Script provides:
   - Direct sheet storage for device data
   - Media file storage in Google Drive

7. CONNECTION & NETWORK SETUP
================================================================================

7.1 APK CONNECTION TO SERVER

The APK connects to the MDM server(s) via REST API over HTTP/HTTPS.

Primary Server URL (build config):
  BuildConfig.BASE_URL = "https://hmdm-server-qlw2.onrender.com"
  
Secondary Server URL (build config):
  BuildConfig.SECONDARY_BASE_URL = "https://hmdmbackend.onrender.com"

Device URL Structure:
  {BASE_URL}/{project}/rest/...

Project path is stored in SettingsHelper (default: "hmdm")

7.2 CONNECTION FLOW
  Step 1: APK starts → reads server URL from SharedPreferences
  Step 2: If no URL configured, shows setup screen
  Step 3: Device enrolls via POST to /rest/public/sync/configuration/{id}
  Step 4: Server assigns device number/ID
  Step 5: APK stores device ID and polls for configuration
  Step 6: Configuration is applied (apps, policies, restrictions)

7.3 NETWORK SECURITY
  - Cleartext traffic permitted (for development/HTTP deployments)
  - System and user CA certificates trusted
  - Optional SSL certificate installation via server URL
  - Request signature verification (configurable via BuildConfig)
  - Unsafe OkHttp client available for self-signed certs

7.4 FAILOVER
  - Primary server failure → attempts secondary server
  - MQTT push failure → falls back to HTTP long polling
  - Config update failure → retry with exponential backoff
  - Offline mode: cached config used until connectivity restored

8. WORKFLOW - APK INSTALL TO PRODUCTION
================================================================================

PHASE 1: FIRST TIME SETUP
  Step 1: Install APK on device
  Step 2: Launch app → InitialSetupActivity appears
  Step 3: App checks for certificates (downloads if URLs provided)
  Step 4: App requests Device Admin (if not already)
  Step 5: Set as Device Owner (if possible) via QR/adb
  Step 6: Enter Server URL (if not pre-configured)
  Step 7: Enter Device ID (auto or manual)
  Step 8: App fetches configuration from server
  Step 9: Policies applied → apps installed → setup complete

PHASE 2: NORMAL OPERATION
  Step 1: App starts → MainActivity loads
  Step 2: Configuration update check (periodic)
  Step 3: Background services start:
     - LocationService (GPS tracking)
     - StatusControlService (policy enforcement)
     - PushLongPollingService or MQTT (notifications)
  Step 4: Device info sent every 15 minutes
  Step 5: Remote logs uploaded as configured
  Step 6: App updates checked and installed automatically

PHASE 3: ADMIN OPERATIONS
  - Remote reboot: trigger from server → APK confirms → device reboots
  - Password reset: trigger from server → APK resets password
  - App install/update: server config update → APK installs silently
  - File push: server config update → APK downloads files
  - Location request: on-demand location push

================================================================================
15. TROUBLESHOOTING & FAQ
================================================================================

Q: APK won't connect to server?
A: Check server URL is correct. Verify network connectivity.
   Check SSL/TLS certificate. Try clearing app data.

Q: Device not enrolling?
A: Ensure server is running. Check device ID is unique.
   Verify server project path matches APK configuration.

Q: Apps not installing?
A: Check app version compatibility. Verify storage permissions.
   Check if unknown sources installation is allowed.

Q: Location not reporting?
A: Verify location permissions are granted (including background).
   Check if GPS is enabled on device. Verify location service is running.

Q: Push notifications not working?
A: Check MQTT service or polling service is running.
   Verify network connectivity. Check firewall settings.

Q: Cannot set as Device Owner?
A: Device must be factory reset first.
   Use adb command: adb shell dpm set-device-owner com.hmdm.launcher/.AdminReceiver
   Device must not have any Google accounts added yet.

Q: Kiosk mode not working?
A: Enable device admin. Restart the app.
   Verify kiosk mode is enabled in server configuration.

Q: How to exit kiosk mode?
A: Enter admin password (if configured). Or use adb to uninstall app.
   Default exit method: tap 5 times on version info in settings.
